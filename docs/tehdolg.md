### 1.5 Медленные / неэффективные средства

- Проблема N+1 запросов в schemas.py при сериализации пациентов. При вызове get_patients SQLAlchemy делает один запрос для получения 100 пациентов, а затем 100 отдельных запросов для подтягивания их активных госпитализаций.
    
- Использование синхронной библиотеки requests в PySide6 блокирует главный поток интерфейса при ожидании ответа от сервера.
    

### 1.6 Незакоммиченный код / долгоживущие ветки

- В репозитории "висит" старая ветка feature/old-gui-prototype, которая не сливалась с main уже 2 месяца, но содержит полезные иконки и ресурсы, которые так и не были перенесены в основную ветку.
    

### 1.7 Отсутствие / несоответствие технической документации

- frontend/mainwindow.py — Документация методов (docstrings) устарела после рефакторинга поиска. Указано использование setBackground(), хотя по факту уже используется setData(Qt.BackgroundRole).

### 1.8 Отсутствие тестовой среды

- Все ручные тесты проводятся на локальной базе данных, которая со временем засорилась тестовыми данными ("Пациент Тест Тестович"). Отсутствует отдельная тестовая БД (test_hims_db), которая очищается перед запуском тестов.
    

### 1.9 Длинные циклы интеграции / отсутствие непрерывной интеграции

- Отсутствует CI/CD пайплайн (например, GitHub Actions или GitLab CI). При каждом изменении кода мне приходится вручную останавливать сервер, активировать venv, устанавливать зависимости и перезапускать uvicorn.

- **Доля технического долга:** ~34% от общего объема.
    

### Выводы и корректировка плана

В норме технический долг не должен превышать 10-15% (Debt Baseline). Текущий показатель в 34% говорит о том, что проект достиг "потолка долга" (Debt Ceiling). Дальнейшая разработка новых фич без рефакторинга приведет к резкому падению скорости разработки и появлению плавающих багов (особенно блокировка UI из-за синхронных запросов).

**Обоснование необходимости устранения:**  
Отсутствие тестовой БД и CI/CD тормозит проверку новых фич. Запутанная архитектура в роутерах делает невозможным написание unit-тестов для бизнес-логики.

**Изменения плана разработки проекта:**

1. Внедрить "чистящий" спринт (Cleanup): отложить разработку модуля PDF-отчетов и расписания врачей на один спринт.
    
2. В текущем спринте посвятить 70% времени (около 16 SP) погашению критической части техдолга (решение проблемы N+1, рефакторинг дублирующегося кода, настройка тестовой БД и RoleChecker).
    
3. В будущих спринтах закладывать строго 15-20% времени на рефакторинг (Definition of Done: "Код чист, тесты проходят"), чтобы график технического долга оставался на уровне естественного минимума.
